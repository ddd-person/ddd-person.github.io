Managed Agents：当 AI 本身成为操作系统

发表于2026-04-14分类于AI应用阅读次数

过去一年，我们看遍了各种 AI Agent 方案：有人在 VS Code 里嵌个 Copilot，有人在终端里跑个 Claude Code，有人搞个浏览器插件让 AI 帮你点按钮……

说白了，这些方案的底层逻辑都一样：在传统操作系统上嫁接 AI。AI 是个外挂，操作系统才是主体。

4 月 8 号，Anthropic 上线了 Managed Agents，思路完全反过来——AI 本身就是操作系统，工具是它的外设。

这个变化有多大？往下看。

一、什么是 Managed Agents？

一句话：Anthropic 把搭建 AI Agent 的所有脏活累活全包了。

你不用自己搭服务器、不用写 Agent 循环、不用管沙箱安全、不用操心状态持久化。你只需要告诉它：用哪个模型、给什么提示词、接哪些工具——然后一个 API 调用，Agent 就在云端跑起来了。

它的核心设计围绕三个概念：

概念作用类比
Agent定义"它是谁"：模型、提示词、工具清单、MCP 连接操作系统的内核配置
Environment定义"它在哪里跑"：容器模板、依赖包、网络规则虚拟机 / 容器镜像
Session定义"这次干什么"：活跃的执行实例，保持上下文和状态一个正在运行的进程

注意这个架构——Agent 是内核，Environment 是硬件抽象层，Session 是进程。这不就是操作系统的结构吗？

二、四个杀手级能力

1. OAuth 一键接入

传统方式接一个外部服务（比如 GitHub、Slack、Jira），你得写 OAuth 回调、存 Token、刷新 Token、处理鉴权失败……一堆模板代码。

Managed Agents 的做法：把凭证存到 Credential Vault（安全保险箱），Agent 运行时通过专用代理自动注入。沙箱本身永远接触不到原始密钥。

换句话说，Agent 能调用你的 GitHub、Slack、数据库，但它拿不到你的密码。安全和便捷，第一次不是二选一。

2. 云端托管——7×24 无人值守

之前跑一个 AI Agent，你的电脑得一直开着，终端不能关，网络不能断。Agent 本质上是你电脑上的一个进程。

Managed Agents 直接跑在 Anthropic 的云上。你给它一个任务，关掉电脑去睡觉，它在云端默默干活。Session 天然持久化，断网了接上还能继续，不会丢状态。

这对需要长时间运行的场景特别关键：

  • 自动化代码审查——PR 提上来，Agent 自己 review 完给意见
  • 数据处理流水线——跑几个小时的 ETL 任务
  • 监控告警——持续观察系统状态，异常时自动处理

3. 沙箱隔离——Agent 不会把你的服务器搞炸

让 AI 在你的服务器上执行代码，你怕不怕？

Managed Agents 的解法是彻底解耦：

  • Harness（驾驭层）：跑 Agent 循环、管上下文、路由工具调用、做错误恢复。无状态，可水平扩展
  • Sandbox（沙箱）：AI 生成的代码、文件操作、Shell 命令在这里执行。完全隔离
  • Session（会话）：持久化记忆和状态，断线重连不丢失

底层揭秘：Sandbox 的状态与隔离机制

很多开发者担心：云端沙箱如果超时回收了，跑了一半的数据会不会丢？

实际上，Managed Agents 的底层设计极其精妙。沙箱的文件系统是与 Session 绑定的持久化快照。当你创建一个 Session 时，系统会为其分配一片专属存储。如果 Agent 在执行 pip install x 或是生成了一堆日志然后突然挂起休眠，只要你继续唤醒这个 Session（在这个 Session 的生命周期内），它所有的文件系统状态依然保留在那儿，仿佛电脑从休眠中醒来一样。

而对于企业极为关心的网络隔离，在创建 Environment 时的 networking="restricted" 参数能直接锁死所有外部出站连接。沙箱内部就成为了一座无法直接访问公网的孤岛。如果需要访问内网的数据库，它只能通过你授权的 MCP 隧道进行通信。这意味着哪怕模型出现幻觉去执行恶意脚本，它也无法向外发散泄露一个比特的数据。

脑子（Harness）和手（Sandbox）是分离的。脑子决策，手执行，手出了问题不影响脑子，脑子也摸不到手里的敏感数据。

4. Code Channels——远程指挥你的 Agent

这是 3 月份上线的能力，跟 Managed Agents 搭配起来威力翻倍。

Code Channels 基于 MCP 协议，让你通过 Telegram、Discord 等消息平台远程操控 Claude Code 会话。架构很巧妙：

  • Channel 插件作为 MCP Server 跑在本地，跟 Claude Code 通过 stdio 通信
  • 外部消息平台只是一个"遥控器"，代码执行始终在本地
  • 三层安全模型：插件白名单 + 配对码认证 + 本地执行

而 Managed Agents 把类似的能力搬到了云端——你不需要本地跑着一台机器了，Agent 在云端执行，你通过 API 随时发消息、查状态、拿结果。

维度Code ChannelsManaged Agents
执行位置你的本地机器Anthropic 云端
适用场景交互式开发，远程操控本地环境生产级，7×24 自动化
状态绑定本地终端会话云端持久化
架构MCP Server 作为本地子进程Harness + Sandbox + Session 解耦

两者不是替代关系，而是互补：开发时用 Code Channels 远程控制，上生产用 Managed Agents 自动运行。

三、对"小龙虾们"的降维打击

这里说的"小龙虾"，是那些在传统操作系统上嫁接 AI 能力的方案——不管是 IDE 插件、终端工具还是浏览器扩展。

它们的共同问题：

问题嫁接式方案Managed Agents
基础设施你自己搭，自己维护Anthropic 全包
状态管理自己写持久化逻辑Session 自动管理
安全隔离自己搭 Docker / 沙箱内置沙箱，凭证保险箱
工具集成每个工具写一套适配代码MCP 协议标准化接入
运行依赖本地得一直开着云端无人值守
可扩展性受限于本机资源云端弹性扩展

打个比方：嫁接式方案就像在自行车上装一个电动马达，能跑是能跑，但本质上它还是自行车。Managed Agents 是直接造了一辆电动车——底座都换了。

当 AI 成为底座、工具成为外设的时候，整个开发范式就变了：

  • 你不再是"用 AI 辅助编程"，而是"指挥 AI 编程"
  • 你不再关心"Agent 怎么跑起来"，而是"Agent 应该做什么"
  • 你不再写胶水代码，而是定义 Agent 的行为规则

四、怎么用？一个 API 调用搞定

上手流程非常简洁：

第一步：创建 Agent

定义模型、提示词和工具：

import anthropic

client = anthropic.Anthropic()

agent = client.agents.create(
    model="claude-sonnet-4-20250514",
    system_prompt="你是一个代码审查助手，负责审查 PR 并给出改进建议。",
    tools=["bash", "file_read", "file_write"],
    mcp_servers=[{
        "name": "github",
        "url": "https://mcp.github.com"
    }]
)

第二步：创建 Environment

定义运行沙箱：

env = client.environments.create(
    packages=["git", "python3", "node"],
    networking="restricted"
)

第三步：启动 Session

跑起来：

session = client.sessions.create(
    agent_id=agent.id,
    environment_id=env.id
)

# 给 Agent 发消息
response = client.sessions.send_event(
    session_id=session.id,
    event={
        "type": "user_message",
        "content": "审查一下 PR #42 的代码变更"
    }
)

三步搞定。不用写 Agent 循环，不用管状态，不用配 Docker。

关键 API 说明

API用途
agents.create创建 Agent 配置（模型、提示词、工具、MCP）
environments.create创建运行环境模板
sessions.create启动一个 Session 实例
sessions.send_event向 Session 发送消息（一次性等待结果）
sessions.stream流式监听 Session 事件（实战推荐）

进阶实战：使用 Stream 流式监听 Agent 执行过程

在生产环境中，你绝大多数时候不会想用 send_event 傻等结果（毕竟 Agent 可能要拉取仓库、运行测试，跑五分钟才结束）。你需要的是实时把执行过程输出给最终用户。这就可以用 client.sessions.stream：

# 实时侦听 Agent 的每一步操作
with client.sessions.stream(
    session_id=session.id,
    event={"type": "user_message", "content": "帮我看看 /src 里的报错"}
) as stream:
    for event in stream:
        if event.type == "tool_use":
            print(f"[{event.tool_name}] 正在执行...")
        elif event.type == "stdout":
            print(f"终端输出: {event.text}")
        elif event.type == "message_delta":
            print(event.text, end="", flush=True)

通过这个流式 API，前端就可以像 ChatGPT 一样把所有的过程日志和工具调用打字机般展示出来，并能在中间环节进行人工介入（Human-in-the-loop）及安全熔断。

注意：使用 API 需要添加 managed-agents-2026-04-01 beta header，官方 SDK 会自动处理。

五、定价

Managed Agents 的定价模型很透明：

  • Token 费用：和标准 Claude API 一样
  • Session 运行时：$0.08 / session-hour（活跃运行时间）

也就是说，Agent 空闲时不计费，只有在实际执行任务时才收费。一个跑了 10 分钟的代码审查任务，Session 费用大约 $0.013——一分多钱人民币。

六、MCP 协议——Agent 的"USB 接口"

Managed Agents 通过 MCP（Model Context Protocol） 接入外部工具，这就像给操作系统装外设一样简单。

MCP 的价值在于标准化：

  • 接 GitHub？配一个 MCP Server
  • 接 Slack？配一个 MCP Server
  • 接你自己的内部 API？写一个 MCP Server

不用为每个工具写一套适配代码，所有外部服务通过统一协议接入。Agent 不需要知道 GitHub API 长什么样，MCP Server 帮它翻译好了。

凭证管理也干净：OAuth Token 和 API Key 存在 Credential Vault 里，运行时由 MCP Proxy 注入，Agent 的沙箱永远碰不到原始密钥。

6.1 实战：5分钟用 FastMCP 暴露本地数据库给 Agent

光说不练假把式。假设你想让跑在云端的 Managed Agent 能查询你内部甚至本地的数据库订单，不用改 Agent 源码，只需用 Python 的 FastMCP 写个几十行的微服务。

from mcp.server.fastmcp import FastMCP
import sqlite3

# 初始化一个名为 "Internal DB Gateway" 的 MCP Server
mcp = FastMCP("Internal DB Gateway")

@mcp.tool()
def get_order_status(order_id: str) -> str:
    """查询内部数据库里的订单状态"""
    conn = sqlite3.connect('local_orders.db')
    cursor = conn.cursor()
    cursor.execute("SELECT status FROM orders WHERE id=?", (order_id,))
    result = cursor.fetchone()
    
    if result:
        return f"订单 {order_id} 状态为: {result[0]}"
    return "未查到该订单"

# 启动 Server
if __name__ == "__main__":
    mcp.run()

把这个微服务跑起来后，即可作为 MCP Server 的载体。在上面第一步创建 Agent 的配置中，把这个 Server 注册进去。你的云端 Agent 瞬间就解锁了安全的内网能力，它会自己判定什么时候调用查询，并完美组装自然语言回复！

七、这意味着什么？

如果你是开发者，Managed Agents 带来的变化是角色的转变：

你不再是一个"写代码的人"，而是一个"定义 Agent 行为的人"。你的工作从"实现逻辑"变成了"设计规则"——告诉 Agent 该做什么、不该做什么、用什么工具、怎么判断成功。

这其实就是之前那篇 Harness Engineering 里讨论的方向：驾驭工程。Managed Agents 就是 Anthropic 给出的官方实现——他们把 Harness 的基础设施标准化了，你只需要关注上层的行为定义。

AI 本身就是操作系统，工具就是它的外设。

这不是一句口号，这是 Managed Agents 的架构实现。当你用 Agent + Environment + Session 三层结构来描述 AI 系统时，传统操作系统里的内核、驱动、进程概念就被完全映射过来了。

而那些还在传统 OS 上嫁接 AI 的方案，就像在 DOS 时代讨论要不要给命令行加个图形界面——不是不行，而是起点就低了一个维度。

九、总结

Managed Agents 不是又一个 Agent 框架，而是 AI 原生的操作系统——它把 Agent 从你电脑上的一个进程，变成了云端永不下线的独立智能体。

对开发者来说，该做的事很明确：

  1. 理解 Agent + Environment + Session 三层抽象
  2. 熟悉 MCP 协议，学会定义工具接入
  3. 把思维从"写代码"转向"定义 Agent 行为"

AI 的操作系统时代来了。别再给自行车装马达了。
